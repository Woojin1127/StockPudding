# TECHSPEC Editor

`docs/refactoring/TECHSPEC.md` 문서를 수정하는 전문 스킬입니다.

> 이 스킬은 사용자와 대화하며 **분석 → 설명 → 승인 → 적용** 흐름을 따릅니다.
> 승인 루프가 필수이므로 반드시 메인 대화 안에서 실행합니다(서브에이전트로 위임 금지).

## Workflow

TECHSPEC 변경 요청 시 반드시 아래 4단계를 순서대로 수행합니다. **어떤 단계도 생략하지 마세요.**

### Phase 1: Codebase Analysis (코드베이스 분석)

변경 요청이 현재 프로젝트에 타당한지 검토합니다.

1. **관련 코드 탐색**
   - 변경 요청과 관련된 디렉토리/파일을 Glob, Grep, Read로 탐색
   - 현재 구현 상태와 요청된 변경 사항 간의 차이점 파악

2. **타당성 평가 기준**
   - 요청된 변경이 현재 코드베이스의 실제 상태를 반영하는가?
   - 변경이 기술적으로 실현 가능한가?
   - 변경이 프로젝트의 기술 스택과 호환되는가?

3. **분석 결과 기록**
   ```
   [분석 대상]: 어떤 파일/디렉토리를 확인했는지
   [현재 상태]: 코드베이스의 현재 구현 상태
   [요청 내용]: 사용자가 요청한 변경
   [타당성]: 타당함 / 부분적으로 타당함 / 타당하지 않음
   [근거]: 판단의 근거
   ```

### Phase 2: Conflict Check (충돌 검사)

변경이 기존 아키텍처, 패턴, 컨벤션과 충돌하지 않는지 확인합니다.

1. **충돌 검사 항목**
   - **아키텍처**: 프로젝트 아키텍처 준수 여부
   - **네이밍 컨벤션**: kebab-case (파일), PascalCase (컴포넌트), camelCase (변수)
   - **Import 패턴**: `@/*` alias 사용, 절대/상대 경로 규칙
   - **디렉토리 구조**: TECHSPEC에 정의된 target structure와의 일관성
   - **기존 TECHSPEC 내용**: 새 내용이 기존 내용과 모순되거나 중복되지 않는지

2. **충돌 발견 시**
   - 충돌 내용을 명확히 기술
   - 해결 방안 제시 (기존 내용 수정, 새 내용 조정, 또는 병합)

### Phase 3: Explain to User (사용자에게 설명)

분석 결과를 사용자에게 **반드시** 설명합니다. 생략 금지.

**설명 포맷:**

```markdown
## TECHSPEC 변경 분석 결과

### 1. 코드베이스 분석

- **탐색한 파일**: [파일 목록]
- **현재 상태**: [요약]
- **타당성**: [타당함/부분적/타당하지 않음] - [이유]

### 2. 충돌 검사

- **아키텍처 충돌**: 없음 / [충돌 내용]
- **컨벤션 충돌**: 없음 / [충돌 내용]
- **기존 TECHSPEC 충돌**: 없음 / [중복 또는 모순 내용]

### 3. 권장 사항

[변경을 진행할지, 수정이 필요한지, 대안이 있는지 등]
```

### Phase 4: Get User Feedback (사용자 피드백 수집)

**필수**: 변경 적용 전에 반드시 1개 이상의 질문으로 사용자 피드백을 받습니다.

1. **질문 유형 예시**
   - "분석 결과를 검토하셨습니까? 이대로 진행해도 될까요?"
   - "충돌이 발견되었는데, [옵션 A]와 [옵션 B] 중 어떤 방식으로 해결할까요?"
   - "[특정 부분]에 대해 추가 설명이 필요하신가요?"
   - "이 변경이 [특정 기능]에 영향을 줄 수 있는데, 괜찮으시겠습니까?"

2. **피드백 확인 후에만 변경 적용**
   - 사용자가 명시적으로 승인한 경우에만 TECHSPEC.md 수정
   - 사용자가 수정을 요청하면 Phase 1로 돌아가 재분석

## Document Writing Rules

TECHSPEC.md 수정 시 준수할 규칙:

1. **한글로 작성**: TECHSPEC.md는 반드시 한글로 작성
2. **Clean Document**: 변경 흔적 남기지 않음 ("수정됨", "변경됨", "~~삭제~~" 금지). 문서는 항상 처음부터 그렇게 작성된 것처럼 깔끔하게 유지
3. **중복 제거**: 새 내용이 기존 내용과 중복되면 단순 추가가 아닌 통합/수정/제거로 탄탄한 문서 유지. 문서가 불필요하게 길어지는 것을 방지
4. **기존 구조 유지 불필요**: 더 나은 구조가 있다면 기존 구조를 재편성 가능

## Change Log

수정 완료 후 `docs/refactoring/TECHSPEC_preference.md`에 변경 로그를 추가합니다.

- 개요는 **한글로 한 줄** 요약하며, 날짜(`[YYYY-MM-DD]`)와 변경 이유/배경을 함께 기록
- 파일이 없으면 새로 생성
- 기존 로그는 유지하고 **맨 아래에 새 항목 추가**
- 사소한 오타 수정이나 포맷팅 변경은 기록하지 않아도 됨

**형식 예시:**

```markdown
- [2026-01-25] `app/` → `src/app/` 이동 추가 - 커뮤니티 best practice 반영
- [2026-01-25] services vs providers 구분 설명 추가 - 사용자 질문에 대한 명확화
```

## Important Files

| File                                      | Purpose                          |
| ----------------------------------------- | -------------------------------- |
| `docs/refactoring/TECHSPEC.md`            | 리팩토링 기술 명세서 (수정 대상) |
| `docs/refactoring/TECHSPEC_preference.md` | 변경 로그 파일                   |
| `.claude/CLAUDE.md`                       | 프로젝트 개요 및 아키텍처 참조   |

## Anti-Patterns (하지 말아야 할 것)

- ❌ 분석 없이 바로 수정
- ❌ 사용자 확인/승인 없이 변경 적용
- ❌ 충돌 검사 생략
- ❌ 한글로 TECHSPEC.md 작성
- ❌ 변경 로그 누락
- ❌ 코드베이스와 맞지 않는 내용 추가
